1. Cross Site Request Forgery  [State: Not Started]  [Priority: High] 

Category:Elevation Of Privilege
Description:Cross-site request forgery (CSRF or XSRF) is a type of attack in which an attacker forces a user's browser to make a forged request to a vulnerable site by exploiting an existing trust relationship between the browser and the vulnerable web site. In a simple scenario, a user is logged in to web site A using a cookie as a credential. The other browses to web site B. Web site B returns a page with a hidden form that posts to web site A. Since the browser will carry the user's cookie to web site A, web site B now can take any action on web site A, for example, adding an admin to an account. The attack can be used to exploit any requests that the browser automatically authenticates, e.g. by session cookie, integrated authentication, IP whitelisting, … The attack can be carried out in many ways such as by luring the victim to a site under control of the attacker, getting the user to click a link in a phishing email, or hacking a reputable web site that the victim will visit. The issue can only be resolved on the server side by requiring that all authenticated state-changing requests include an additional piece of secret payload (canary or CSRF token) which is known only to the legitimate web site and the browser and which is protected in transit through SSL/TLS. See the Forgery Protection property on the flow stencil for a list of mitigations.
Justification: Jenkins sanitizes form data by filtering special characters. When the request is send the form is validated using strong encryption and strong tokens.

2. Elevation by Changing the Execution Flow in Jenkins Master Web Service  [State: Not Started]  [Priority: High] 

Category:Elevation Of Privilege
Description:An attacker may pass data into Jenkins Master Web Service in order to change the flow of program execution within Jenkins Master Web Service to the attacker's choosing.
Justification: This occurs when the attacker is able to gain elevated rights through unauthorized means. The threat is to subvert any authentication/its mechanisms. This might be mitigated through appropriate input validation and exception handling. The application can also guard against buffer overflows and other memory management issues. Also, have the application default to the least of the person's privileges. One can also protect authorization paths, i.e. protect code paths that interact with the database in order to ensure the authorization and authentication process.

3. Jenkins Master Web Service May be Subject to Elevation of Privilege Using Remote Code Execution  [State: Not Started]  [Priority: High] 

Category:Elevation Of Privilege
Description:Browser may be able to remotely execute code for Jenkins Master Web Service.
Justification: A DoS may cause a process or a datastore to be unable to process incoming requests and make the application unusable. If a data flow is interrupted by an internal server error or a denial of service, then customers might lose information. The mitigation is to require authentication by requesting entities. Also, limit the number of retries for authentication. One could also increase the size of the TCP connection queue in order to handle more requests. One could also decrease the connection establishment period. You could also use a dynamic queue so that it does not fill up. Hopefully a system that performs intrustion detection can be implemented on the network to respond to SYN attacks.

4. Data Flow HTTPS request Is Potentially Interrupted  [State: Not Started]  [Priority: High] 

Category:Denial Of Service
Description:An external agent interrupts data flowing across a trust boundary in either direction.
Justification: The attack might be able to legitimately prevent users from accessing an application and interrupt data across a trust boundary. The mitigation is mainly within the network. The network should not block waiting for responses that cross the trust boundary. If possible, implement the network so that it continues data processing, but at a reduced capacity. If it ceases to proces requests, it should provide appropriate responses to waiting users. The big points to this protective strategy are: denial of service protection, boundary protection, and transmission Confidentiality/integrity. In May of 2015 Jenkins actually had a collision DoS vulnerability. This was a bundled version of commons-fileupload library and Jenkins backported to the previous version bundled with Jenkins. One possible plug is that Jenkins users must login before they access most of the Jenkins functionality. This security feature helps reduce the possiblity of DoS attacks. Also, the servers are often hosted by the client, so this specific security fix might be implemented by its users.

5. Potential Process Crash or Stop for Jenkins Master Web Service  [State: Not Started]  [Priority: High] 

Category:Denial Of Service
Description:Jenkins Master Web Service crashes, halts, stops or runs slowly; in all cases violating an availability metric.
Justification: Jenkins can be isolated and run behind a company firewall. Services can also be queued and processed at a later time by Jenkins. It is usually expected that the processes run by Jenkins will be large and take some time for the build and testing process. In this way, the queue makes Jenkins functional for later and it still seems usable.

6. Potential Data Repudiation by Jenkins Master Web Service  [State: Not Started]  [Priority: High] 

Category:Repudiation
Description:Jenkins Master Web Service claims that it did not receive data from a source outside the trust boundary. Consider using logging or auditing to record the source, time, and summary of the received data.
Justification: When running Jenkins locally, it outputs all default logging information to standard out. Jenkins also has it setup where logs will be made available on its system (depending on the OS). Jenkins uses java.util.logging for logging. Jenkins also bundles the logs from the entire JVM into a single place. Jenkins is event more usable and provides a GUI for configuring/collecting/reporting log information. The logging records errors, such as not receiving data. The user can record logs when auditing records or data.
